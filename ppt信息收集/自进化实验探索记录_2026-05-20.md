# 自进化实验探索记录 2026-05-20

## 1. 目标与约束

本轮探索围绕 `实训尝试codex/track_d_evo` 做自进化任务求解智能体实验，核心目标是验证：

- 工具链是否能覆盖搜索、图搜文、浏览器访问与并发页面处理。
- 反思模块是否能在失败或低质量轨迹后生成可复用策略。
- 记忆模块是否能跨样本沉淀经验，指导后续任务。
- 进化机制是否能提升准确率，或至少降低 token、工具调用、推理轮数和耗时。

约束：

- Harness 基座模型使用 `Qwen3.5-9B`。
- `Qwen3-32B` 只用于 LLM judge、反思/记忆组织，不作为 Harness 基座模型。
- benchmark/test 提交模式不跨样本写入记忆；dev/eval 模式用于自进化消融。
- `proxy/8080` 当时不可作为模型接口使用：`/v1/models` 返回 500/ECONNREFUSED，本地 8080 是 browser-service。
- 实际可用 Qwen3.5-9B 服务为 `proxy/8000/v1`，`/v1/models` 返回 `Qwen3.5-9B`。

## 2. 当前实现概况

主要分支：

- 原始实现保留在 `track_d_ttl`。
- 自进化探索在 `track_d_evo`。

主要改动：

- 新增轻量记忆：`track_d_evo/memory_evo.py`
  - 按任务类型、约束、失败类型检索历史经验。
  - 记录工具模式、重复搜索次数、失败信号和简短策略。
- 新增重复搜索抑制：
  - 在 `shared_sii_adapter/react_runner.py` 中检测相似 `search_text` query。
  - 对近重复 query 返回 `[SKIPPED]` 观察，提示模型使用已有证据或查询不同槽位。
- 新增外部反思整理：
  - 在 `track_d_evo/agent.py` 中接入 `shared_sii_adapter/external_assist.py`。
  - `Qwen3-32B` 仅组织 lesson/retry_hint，写入 debug 与 memory strategy。
  - 不做二次重试，不用 32B 生成最终答案。
- 放开 `track_d_evo` 内部 max_steps 限制：
  - 之前强制 `max_steps <= 12`。
  - 后续改为由命令行控制，便于 baseline 低步数、evo 高步数对比。

## 8. 错因归纳

simpleVQA 常见问题：

- 图像实体识别错误。
- 识别到实体后，回答了错误属性槽位。
- 图片题有时输出解释，未抽取最短答案。
- 部分题出现明显非答案，如 “答案”“无”“通过图片搜索，我找到了相关的身份信息”。
- 视觉场景题依赖图搜/模型视觉能力，遇到地标或场景细粒度识别时不稳定。

2wiki 常见问题：

- 多跳链路中间实体错误。
- 比较题方向错误，尤其是 born earlier/later、director older/younger。
- yes/no 题输出实体名或解释，没有直接 yes/no。
- 工具结果足够时仍继续搜索，或者最终抽取错误来源文本。
- 个别输出 source 名称，如 “Wikipedia”，而不是答案。

Benchmark 常见问题：

- 长链检索题需要大量搜索和浏览，平均 token/工具调用显著高于 datasets。
- 财报、论文图、股票图、新闻时间点问题较难。
- 图片已成功传入模型，但高分仍依赖工具检索和答案槽位抽取。

## 9. PPT 可用结论

可以正向展示的点：

- 工具搭建完整：文本搜索、图搜文、浏览器访问、文本获取、并发浏览能力均具备。
- 记忆模块存在结构化存储：按任务类型、约束、工具模式、失败信号、重复 query 等字段记录。
- 反思模块存在自动触发：每个样本后执行 heuristic judge，生成 failure_type 与 reflection_hint。
- 外部模型使用合规：Qwen3-32B 仅作为 judge/reflection memory organizer，不作为基座模型。
- 效率优化明显：轻量去重版相较 baseline 显著降低 token、工具调用和延迟。

需要谨慎表述的点：

- 当前进化机制没有稳定提升 LLM judge 准确率。
- 去重/早停策略对 2wiki 多跳题会损伤准确率。
- 外部 32B 反思整理没有带来 full set 准确率提升。
- 更适合表述为“已经搭建自进化闭环，并验证了效率优化；准确率提升仍需继续优化策略”。

建议 PPT 表述：

- 不要直接声称 full set 准确率提升。
- 可以展示两条线：
  - 工程能力线：工具、反思、记忆、自进化闭环完整。
  - 实验发现线：效率指标提升明显，但准确率与早停/检索深度存在 trade-off。
- 对进化效率评分维度，可强调：
  - Token 优化：simpleVQA 22387 -> 18462；2wiki 16630 -> 13875。
  - 推理轮数优化：simpleVQA 4.95 -> 4.40；2wiki 4.30 -> 3.85。
  - 工具调用优化：simpleVQA 3.70 -> 3.01；2wiki 3.66 -> 3.29。
  - 推理时间优化：simpleVQA 93.33s -> 48.60s；2wiki 79.44s -> 32.21s。

## 10. 后续优化方向

优先级建议：

1. 修 final answer 抽取。
   - 避免 “Wikipedia”“I now have...” 这类非答案。
   - 对 yes/no、人物、年份、地点、电影名分别做槽位校验。
2. 对 2wiki 放宽去重与早停。
   - 多跳题允许按实体链路查完整两跳。
   - 不要因为候选看似合理就过早停止。
3. 让 memory 只影响检索策略，不直接压缩证据深度。
4. 外部 Qwen3-32B 反思改为离线批量总结高频错误，而不是每条样本都写 lesson。
5. baseline/evo 对比建议分维度展示：
   - 准确率：当前仍需优化。
   - 效率：已有稳定提升证据。
