## 1. 项目背景

本项目围绕“自进化的任务求解智能体”课题展开，目标是在给定 Harness 框架下构建可运行、可评测、可提交的 Agent 系统，并探索 Reflection、Memory、Tool Calling、LLM-as-a-Judge 等机制对任务求解能力和推理效率的影响。

主要评测对象：

- `2wiki`：多跳问答任务，强调检索、实体链路和推理。
- `simpleVQA`：视觉问答任务，强调图片理解、多模态输入和实体识别。
- `benchmark.csv`：正式打榜数据，要求按顺序作答并提交答案与轨迹。

## 3. 合规边界

根据 `要求及回复.md`、`Readme.md`、课题 PDF 和老师回复，本次探索遵循以下边界：

- 正式打榜数据集不允许 test-time evolution。
- 正式打榜数据集允许 LLM-as-a-Judge，但不能把 gold answer、label、ground truth 输入 Judge。
- 测试时不允许直接用模型输出与 gold answer 比较来判断失败。
- 测试集整体只允许跑一次，处理过的 case 不允许回头重跑。
- 允许在单个样本内部基于低置信、无答案、证据不足等信号触发重试。
- 允许在 `2wiki` 和 `SimpleVQA` 测试集问题上做自蒸馏参数微调。
- 不允许在正式打榜数据上做自蒸馏参数微调。
- `Qwen3-32B` 只能用于 Judge、反思和记忆组织，不能作为 Harness 基座模型直接答题。
- 训练集可以自由选用，但不能与测试集重合，不能包含测试题答案泄露。

PPT 表述建议：

```text
本项目严格区分“验证集自进化探索”和“正式打榜提交”。验证集用于分析 Reflection/Memory/Judge 的效果，正式打榜阶段不进行跨样本测试时进化。
```

## 4. 模型与工具服务

基础作答模型：

```text
Qwen3.5-9B
```

用途：

- Harness 基座模型。
- 负责直接回答问题、读取文本、处理多模态输入。

32B 辅助模型：

```text
Qwen3-32B
```

用途：

- LLM-as-a-Judge。
- 反思整理。
- 记忆组织。

限制：

- 不作为最终答题模型。
- 不接收 gold answer。
- 输出中的 `<think>` 需要清洗，不能直接当作 JSON 或最终答案。

工具服务：

- 浏览器服务：用于页面访问、文本抽取、点击、输入和并发浏览。
- 搜索代理：用于文本搜索、图片搜索、Jina 页面读取等。
- GPU 模型服务：通过 Notebook proxy 暴露 OpenAI-compatible API。

已处理过的问题：

- 搜索代理环境变量中混入零宽字符会触发 `UnicodeEncodeError: latin-1`，已在工具代码中增加清洗。
- 多模态 benchmark 中图片字段不是 URL，而是裸 base64，不能直接当作图搜 URL。
- 图片格式包括 PNG、JPEG、WebP，代码已按 base64 头自动识别 MIME 类型。
- CPU 机器不能直接访问另一个 Notebook 内部 localhost 端口，必须走平台 proxy URL。

## 5. 数据集与多模态发现

正式 benchmark：

- 共 100 条。
- 约 50 条纯文本题。
- 约 50 条多模态题。
- 多模态图片字段是裸 base64，不是在线 URL。

验证集：

- `datasets/2wiki.jsonl`
- `datasets/simpleVQA/SimpleVQA.jsonl`

SimpleVQA 处理方式：

- 同时传入本地图片 base64 和远程 `image_url`。
- 模型可以直接看图。
- 开启工具时也可以调用图搜。

关键发现：

- 多模态题不能只依赖搜索工具。
- 若没有正确把 base64 图片传入模型，模型会误以为缺少图片并尝试搜索，导致答案不稳定。
- 当前主要瓶颈之一是 9B 模型对视觉实体、地标、场景和细粒度属性识别不稳定。

## 7. 自进化模块设计

Reflection：

- 在失败、低置信、无答案、工具循环、证据不足等情况下触发。
- 目标不是简单重试，而是定位失败原因并生成修正策略。
- 反思结果可以转化为后续任务可复用的经验。

Memory：

- 保存历史样本中的策略经验。
- 记录任务类型、失败模式、工具使用方式和改进建议。
- 后续样本构造 prompt 时检索并注入相关经验。
- 记忆不能包含正式测试集 gold answer。

LLM-as-a-Judge：

- 使用 Qwen3-32B 判断当前轨迹和预测是否足够可信。
- Judge 输入包含问题、模型预测、推理路径和已有记忆。
- Judge 输入不包含 gold answer。
- Judge 输出 JSON，包含 pass、reason、memory、retry_hint 等字段。

工具调用：

- 支持文本搜索、图片搜索、浏览器访问、页面文本提取、并发浏览、点击和输入。
- 增加重复搜索抑制，减少相似 query 造成的 token 和时间浪费。

## 9. 进化实验结果摘要

在 `实训尝试codex/track_d_evo` 中做过更完整的 baseline/evolution 对比。

Full baseline：

| 数据集 | n | strict acc | LLM judge acc | 平均 token | 平均工具调用 | 平均延迟 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| simpleVQA | 99 | 30.30% | 69.70% | 22387.00 | 3.70 | 93.33s |
| 2wiki | 100 | 44.00% | 80.00% | 16630.01 | 3.66 | 79.44s |

轻量进化去重版：

| 数据集 | n | strict acc | LLM judge acc | 平均 token | 平均工具调用 | 平均延迟 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| simpleVQA | 99 | 30.30% | 66.67% | 18461.96 | 3.01 | 48.60s |
| 2wiki | 100 | 33.00% | 70.00% | 13875.10 | 3.29 | 32.21s |

观察：

- 轻量进化去重版显著降低 token、工具调用和延迟。
- 准确率和 LLM judge acc 有下降。
- 当前更适合作为“效率优化消融”结果，而不是最终准确率提升结果。

外部反思增强版：

| 数据集 | n | strict acc | LLM judge acc | 平均 token | 平均工具调用 | 平均延迟 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| simpleVQA | 99 | 26.26% | 59.60% | 17919.74 | 2.87 | 50.60s |
| 2wiki | 100 | 33.00% | 71.00% | 17111.12 | 3.70 | 34.07s |

观察：

- 外部 32B 反思确实写入了 debug/memory。
- Full 准确率没有提升。
- 输出格式问题仍较突出，例如输出解释、输出 “Wikipedia”、输出 “答案” 或 “无”。

## 12. 错因归纳

SimpleVQA：

- 图片实体识别错误。
- 地标、人物、Logo、场景细粒度判断不稳定。
- 识别到实体后，回答了错误属性槽位。
- 输出解释而不是短答案。
- 部分样本空输出。

2Wiki：

- 多跳链路中的中间实体错误。
- 比较题方向错误，例如 earlier/later、older/younger。
- yes/no 题没有直接输出 yes/no。
- 证据足够时仍继续搜索，造成答案漂移。
- 最终答案抽取不稳，有时输出来源名而不是答案。

Benchmark：

- 长链检索题平均 token 和工具调用显著增加。
- 财报、论文图、股票图、新闻时间点问题较难。
- 多模态图片虽然能传入模型，但仍需要更强的视觉识别和外部检索配合。

## 14. 后续优化方向

优先级较高：

- 强化最终答案抽取，约束输出为短答案。
- 对 yes/no、比较题、实体属性题加入任务类型模板。
- 对 SimpleVQA 的图片实体识别引入更稳的图搜或视觉辅助策略。
- 对空输出样本做单样本内 Judge 触发重试，但不能使用 gold。
- 优化 Memory 结构，让记忆记录“题型 -> 失败模式 -> 工具策略 -> 答案格式”。

实验设计：

- baseline 和 evolution 需要保证同一任务集合、同一参数、同一运行条件。
- 正式打榜只跑一次，不回头重跑已处理 case。
- 验证集可用于 self-evolution 对比，但要清楚区分验证集和打榜集。

PPT 建议章节：

- 任务背景与合规边界。
- 系统架构：Base Model、Tools、Judge、Reflection、Memory。
- 多模态处理：base64 图片与 MIME 识别。
- 实验设计：baseline、evolved、消融。
- 结果分析：准确率、LLM judge、token、工具调用、延迟。
- 失败案例与改进方向。
- 工程产物：GitHub、HF、提交包、复现命令。
