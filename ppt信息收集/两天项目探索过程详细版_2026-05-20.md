# 两天项目探索过程详细版

更新时间：2026-05-20

本文用于后续制作 PPT，按“两天探索过程”梳理本项目从环境理解、baseline 搭建、多路线实现、自进化实验、蒸馏训练、benchmark 提交到最终反思的完整过程。

## 1. 一句话概括

这两天的探索主线是：先把课程给定的自进化智能体任务跑通，再围绕 Track-D 工具调用 Agent 搭建可复现的评测闭环，随后并行探索 Reflection、Memory、Tool Calling、LLM-as-a-Judge、Planner、Answer Slot、Distillation/SFT 等路线，最终形成了可提交代码包、可展示实验结果、可上传模型产物和后续优化方向。

## 2. 项目背景与目标理解

项目主题是“自进化的任务求解智能体”。核心不是单纯问答，而是构建一个可以在 Harness 框架下进行工具调用、搜索、浏览、反思、记忆和评测的 Agent。

主要任务形态包括：

| 数据 / 任务 | 特点 | 对 Agent 的要求 |
| --- | --- | --- |
| `2wiki` | 文本多跳问答 | 实体链路、时间比较、关系推理、证据检索 |
| `simpleVQA` | 多模态视觉问答 | 看图、实体识别、图搜、视觉锚点、短答案抽取 |
| `benchmark.csv` | 正式打榜数据 | 100 条样本，包含文本与图片题，需要生成提交文件和轨迹 |

最初需要解决的关键问题：

- 如何在平台目录内保存代码、模型、数据和运行结果。
- 如何正确使用 Qwen3.5-9B 作为 Harness 基座模型。
- 如何合规使用 Qwen3-32B，只用于 judge、reflection、memory organization，不能直接作为答题基座。
- 如何处理正式 benchmark 与验证集自进化的合规边界。
- 如何把工具调用轨迹保存下来，为后续 SFT / DPO / 蒸馏做准备。

## 3. 合规边界确认

探索早期先明确了哪些可以做、哪些不能做：

- 正式 benchmark / leaderboard 数据不能跨样本做 test-time evolution。
- 正式 benchmark 允许 LLM-as-a-Judge，但不能给 judge 输入 gold answer。
- 测试集整体只允许单次处理，不能把跑过的样本结果反复用于更新再重跑。
- 验证集和公开 train/dev 数据可以用于自进化探索、消融实验和蒸馏数据构造。
- Qwen3-32B 可以用于 judge、反思、记忆整理、教师数据生成，但不能作为正式 Harness base model。
- 记忆模块不能保存 gold answer，只能保存失败类型、策略、工具模式、经验等无答案泄漏信息。

PPT 中建议表述：

> 我们将“验证集自进化探索”和“正式 benchmark 提交”严格隔离：验证集用于研究 Reflection/Memory/Judge 的效果，正式提交阶段关闭跨样本进化更新，避免测试集信息泄漏。

### 4.3 发现 benchmark 图片字段是裸 base64

早期一个关键发现是：benchmark 的图片字段不是 URL，而是裸 base64。若直接按 URL 处理，模型会误判“没有图片”，然后转向图搜或文本搜索，导致答案漂移。

为此进行了两类修复：

- 将 base64 图片直接作为多模态输入传给模型。
- 按 base64 header 自动推断 `image/png`、`image/jpeg`、`image/webp`。

这个发现对 simpleVQA 和 benchmark 图片题非常关键。

## 6. 第二天：Track-D 工具调用 Agent 深挖

### 6.1 为什么聚焦 Track-D

多路线初步搭建后，核心转向 Track-D，原因是：

- Track-D 和 benchmark 的工具调用要求最直接相关。
- 它能保留真实搜索、浏览、图搜轨迹，适合后续 SFT / DPO。
- 它能体现 Reflection、Memory、Tool Calling、Judge 等评分关注点。
- 当前模型 Qwen3.5-9B 能通过工具弥补知识和视觉识别不足。

### 6.2 Prompt3 / XMLFix 基线

历史最佳正常版是：

```text
runs/benchmark/final_group8_track_d/
```

结果：

| 版本 | Strict | Qwen3-32B LLM Judge |
| --- | ---: | ---: |
| `final_group8_track_d` | 30% | 44% |

该版本特点：

- system prompt 较短。
- 明确 7 类工具白名单。
- 禁止模型伪造 `<tool_call>` 或 `<function=...>`。
- 每一步只能真实调用工具或给最终答案。
- 对图片题和多跳文本题给出基础搜索策略。
- 加入 XML fallback，避免把工具调用文本当成最终答案。

复盘结论：

- 短 prompt + 工具白名单 + fallback 是当时最稳的组合。
- 同配置复跑会波动，说明搜索结果、网页状态和模型采样都会影响结果。

### 6.3 长规则 prompt 的负结果

尝试过更复杂的 hypothesis / candidate / uncertainty prompt，加入：

- 高信息增益搜索。
- 候选生命周期管理。
- browser 状态污染检测。
- 停止条件。
- 不确定性控制。

结果：

| 实验 | Strict | LLM Judge |
| --- | ---: | ---: |
| `qwen35_9b_hypothesis_softguard_c100_ref` | 19% | 29% |

结论：

Qwen3.5-9B 对很长、很复杂的规则 prompt 执行不稳定。规则堆得越多，模型越容易机械执行、过度搜索、输出解释或忽略最终答案槽位。

后续方向因此转为：把策略外置为 planner / policy / skill，而不是继续加长 system prompt。

### 6.4 Slot Planner 探索

在 `track_d_ttl/planner.py` 中加入 deterministic slot planner。

作用：

- 从问题中推断 `answer_slot`，例如人物、年份、地点、yes/no、作品名。
- 提取高/中/低信息约束。
- 生成 query seed。
- 将 planner context 注入 prompt。

结果：

| 实验 | Strict | LLM Judge |
| --- | ---: | ---: |
| `qwen35_9b_slotplanner_c100_ref` | 30% | 39% |
| `qwen35_9b_slotplanner_rerun2_c100_ref` | 25% | 35% |

结论：

- slot planner 对 strict 有帮助，能达到 30%。
- 但 LLM judge 不如历史 prompt3 的 44%。
- 规则抽取仍有噪声，例如把指代词或低价值词误当作 high constraint。

## 7. 第二天：Track-D Evo 自进化实验

### 7.1 Track D Evo 设计

路径：

```text
实训尝试codex/track_d_evo/
```

这是从 `track_d_ttl` 复制出来的自进化实验分支，避免污染原始 Track-D。

核心模块：

| 文件 | 作用 |
| --- | --- |
| `agent.py` | 接入 reflection、memory、external assist |
| `memory_evo.py` | run-scoped 结构化记忆 |
| `skills.py` | 可复用技能提示 |
| `judge.py` | 无 gold 失败信号判断 |
| `refine.py` | 失败到策略的转化 |

设计目标：

- 失败后自动触发 reflection。
- 将失败归因为 no answer、tool-call leak、uncertainty、insufficient evidence、image evidence missing、entity drift、temporal drift 等类型。
- 将失败经验写入 run 内 memory。
- 后续样本根据任务类型、槽位、失败模式检索相关经验。
- Qwen3-32B 可作为 external assistant 整理 lesson / retry_hint，但不能生成最终答案。

### 7.2 Full baseline 结果

目录：

```text
runs/baseline_qwen35_8000_v1/
```

配置：

- Base：Qwen3.5-9B。
- Judge：Qwen3-32B。
- 关闭 reflection、memory、evolution updates。
- `max_steps=12`，`max_tokens=8000`。

结果：

| 数据集 | n | Strict | LLM Judge | Avg Tokens | Avg Turns | Avg Tools | Avg Latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| simpleVQA | 99 | 30.30% | 69.70% | 22387.00 | 4.95 | 3.70 | 93.33s |
| 2wiki | 100 | 44.00% | 80.00% | 16630.01 | 4.30 | 3.66 | 79.44s |

### 7.3 轻量去重 evolution 结果

目录：

```text
runs/evolution_full_dedupe_v1/
```

机制：

- 轻量 memory。
- near-duplicate `search_text` 抑制。
- 尽早停止，减少重复搜索。

结果：

| 数据集 | n | Strict | LLM Judge | Avg Tokens | Avg Turns | Avg Tools | Avg Latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| simpleVQA | 99 | 30.30% | 66.67% | 18461.96 | 4.40 | 3.01 | 48.60s |
| 2wiki | 100 | 33.00% | 70.00% | 13875.10 | 3.85 | 3.29 | 32.21s |

探索结论：

- 效率明显提升：token、工具调用、轮数、耗时都下降。
- 但准确率下降，尤其 2wiki 从 80% LLM Judge 降到 70%。
- 说明多跳问答不适合过早停止，去重和早停会减少必要证据。

PPT 可将其作为“效率优化消融”，不能表述成准确率提升。

### 7.4 外部 32B 反思增强

在 `track_d_evo/agent.py` 中接入：

```text
shared_sii_adapter/external_assist.py
```

Qwen3-32B 的用途：

- 只整理 lesson / retry_hint。
- 写入 debug 和 memory strategy。
- 不参与最终作答。
- 不替代 Qwen3.5-9B。

结果显示：

- external assist 确实写入了记录。
- 但 full set 准确率没有稳定提升。
- simpleVQA 下降明显，2wiki 从 70% 小幅到 71%，仍低于 baseline。

阶段性判断：

外部反思如果只写 memory、不给当前样本二次修正，对当前样本收益有限；如果 memory 提示不够精确，还可能干扰视觉题的实体识别。

## 8. 第二天：Track D Slot Evo 强化

### 8.1 为什么新增 Slot Evo

前面的 Track D Evo 暴露出一个核心问题：模型不只是“找不到证据”，还经常“答错槽位”。

典型错误：

- 问年份，输出完整解释。
- 问人物，输出机构或网页名。
- yes/no 题输出实体名。
- 图片题识别到候选后，回答了错误属性。
- 最终答案出现 `Wikipedia`、`I now have all the information I need`、`答案`、`无` 等非答案。

因此新增：

```text
实训尝试codex/track_d_slot_evo/
```

### 8.2 Slot Evo 模块

核心文件：

| 文件 | 作用 |
| --- | --- |
| `agent.py` | 注入证据规划、skills、memory、reflection、debug |
| `skills.py` | entity lock、temporal lock、slot answer 等技能 |
| `memory_evo.py` | 结构化记忆，按任务类型/槽位/失败模式检索 |
| `policy.py` | 检测低可信源、重复 query、时间约束遗漏、工具失败 fallback |
| `judge.py` | 本地启发式 judge |
| `refine.py` | 将失败信号转为可复用策略 |

新增技能包括：

- `entity_lock`：图像题先锁定图中实体。
- `temporal_lock`：保留 before / after / first / season / as-of 等时间约束。
- `slot_answer`：最终答案只填目标槽位。
- `visual_direct_answer`：视觉关系题优先看图，不被搜索带偏。
- `visual_fact_anchor`：先提取图片可见锚点，再查属性。
- `cross_check_candidate`：多候选时做交叉验证。
- `tool_failure_recovery`：工具失败后换源或使用 snippet。

### 8.3 Slot Evo 结果

2wiki：

| 设置 | n | Strict | LLM Judge | Avg Tokens | Avg Turns | Avg Tools | Avg Latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| no-evo baseline | 100 | 42.00% | 74.00% | 29788.03 | 5.11 | 4.43 | 37.93s |
| slot evo v6 | 100 | 53.00% | 79.00% | 23821.22 | 4.60 | 3.79 | 34.96s |

simpleVQA：

| 设置 | n | Strict | LLM Judge | Avg Tokens | Avg Turns | Avg Tools | Avg Latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| no-evo baseline | 99 | 28.28% | 65.66% | 31374.32 | 5.70 | 4.45 | 67.23s |
| slot evo v6 | 99 | 31.31% | 63.64% | 26043.36 | 4.90 | 3.38 | 49.13s |
| slot evo v7 full fast | 99 | 32.32% | 65.66% | 33033.85 | 5.17 | 3.90 | 63.95s |

结论：

- 2wiki 上效果最清晰：strict 从 42% 到 53%，LLM Judge 从 74% 到 79%，同时 token、轮数、工具调用、耗时下降。
- simpleVQA 上 strict 有提升，LLM Judge 与 baseline 持平或略低，说明视觉实体识别仍是瓶颈。
- Slot Evo 比简单去重更合理，因为它不是单纯减少工具，而是围绕答案槽位和失败类型做策略约束。

PPT 中建议将 2wiki Slot Evo 作为主正向结果，把 simpleVQA 作为“视觉任务仍需继续优化”的分析结果。

## 9. 第二天：蒸馏与 SFT 路线

### 9.1 蒸馏目标

蒸馏路线目标是让 Qwen3.5-9B 学会 Track-D 风格的工具调用过程，而不是只学习最终答案。

设定：

- Student：多模态 Qwen3.5-9B。
- Text teacher：Qwen3-32B。
- Vision teacher：Qwen2.5-VL-32B-Instruct。
- 数据形式：真实 Track-D trajectory。
- 训练方式：黑盒蒸馏，学习 teacher 的输入输出轨迹，不访问 teacher logits。

## 11. 关键失败模式复盘

### 11.1 工具服务不稳定

一次 student benchmark 异常低分定位到工具服务问题：

- `search_text` 错误率曾达到 97.3%。
- 总 tool 错误率达到 80.3%。
- 主要错误是 `127.0.0.1:1227 Connection refused`。

对比历史较好运行：

- `search_text` 错误率约 0.5%。
- 总 tool 错误率约 6.1%。

结论：

工具调用 Agent 的分数高度依赖 search-proxy、browser-service、模型 endpoint 的健康状态。跑 benchmark 或蒸馏采集前必须先做工具健康检查。

### 11.2 2wiki 多跳题不适合过早停止

2wiki 需要完整实体链和比较链。轻量去重虽然减少 token 和工具调用，但容易损伤准确率：

- 中间实体没有查全。
- 时间比较方向错误。
- source 名称被误当答案。
- yes/no 题没有输出 yes/no。

### 11.3 simpleVQA 瓶颈在视觉实体识别

simpleVQA 的主要错误：

- 图像实体识别错误。
- 图搜候选漂移。
- 识别到实体后回答了错误属性。
- 视觉关系题过度搜索，被网页结果带偏。
- 最终答案过长或不是目标槽位。

说明视觉题不能只靠搜索策略，仍需要更强的视觉 grounding 和更稳定的图像实体锁定。

### 11.4 长 prompt 不适合小模型

Qwen3.5-9B 对长规则 prompt 的执行稳定性不佳。更有效的方向是：

- prompt 保持短而明确。
- 策略外置到 planner / policy / skill。
- 用程序保证槽位、工具、去重和收尾，而不是把所有规则塞给模型。
