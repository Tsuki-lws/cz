# 自进化与蒸馏探索基础信息

更新时间：2026-05-20

本文用于 PPT 前期资料收集，整理当前对话中已经完成或验证过的关键探索，包括蒸馏路线、训练环境、Track-D benchmark、自进化模块设计、实验结果和可展示结论。

## 1. 总体目标

当前实训方向围绕 Track-D 工具调用 Agent 展开，主要目标是：

1. 使用老师模型生成轨迹数据，对多模态 student 模型进行蒸馏。
2. student 模型为多模态 Qwen3.5-9B，路径固定为：
   `/inspire/qb-ilm2/project/26summer-camp-01/public/Qwen3.5-9B`
3. 基于 Track-D harness 扩展自进化能力，包括反思、记忆、策略更新和工具调用优化。
4. 分别在 `2wiki` 和 `SimpleVQA` 两类数据集上评估 baseline 与 evo 的准确率、token 消耗、轮数、工具调用数和耗时。

## 2. 数据集与任务形态

当前重点使用两个验证数据集：

| 数据集 | 路径 | 样本数 | 任务特点 |
| --- | --- | ---: | --- |
| 2wiki | `/inspire/qb-ilm2/project/26summer-camp-01/26210300/datasets/2wiki.jsonl` | 100 | 文本多跳问答，实体关系、时间约束、比较推理较多 |
| SimpleVQA | `/inspire/qb-ilm2/project/26summer-camp-01/26210300/datasets/simpleVQA/SimpleVQA.jsonl` | 99 | 多模态图像问答，包含地标、人物、书籍、电影海报、空间关系、类别、年份、数量等 |

结论：这两个数据集题型差异明显，应该分开进化、分开维护记忆。2wiki 更适合实体链、时间链和证据图策略；SimpleVQA 更依赖图像实体锁定、视觉锚点和答案槽位控制。

## 3. 蒸馏路线

前期判断当前蒸馏形式更接近黑盒蒸馏：

- 通过 teacher API 或 teacher 服务生成回答轨迹。
- student 学习 teacher 的输入输出和工具使用轨迹，而不是直接访问 teacher logits。
- 支持文本 teacher 和视觉 teacher，但 student 是多模态模型，因此训练数据和 prompt 要保留图像输入能力。

蒸馏流程脚本和配置集中在：

- `实训尝试codex/distill/scripts/`
- `实训尝试codex/distill/configs/`
- `实训尝试codex/qwen35_sft_multimodal.yaml`
- `实训尝试codex/train_qwen35_sft.sh`

训练曾使用 LLaMA-Factory，环境为 `pegp`，模型为 Qwen3.5-9B 多模态模型。训练中遇到过：

- `llamafactory-cli not found`：通过在 `pegp` 环境安装或 source 环境解决。
- DeepSpeed 配置 JSON 格式错误：需要保证 deepspeed 配置为合法 JSON。
- `flash_attn_func` 导入错误：切到 SDPA 或处理 flash-attn 版本。
- torch 2.9 与 Conv3D 组合被 LLaMA-Factory 拦截：需要降级 torch 到 `<2.9` 或避免触发该组合。

## 5. Track-D Benchmark 现象

早期 Track-D benchmark 中观察到的问题：

1. reasoning 内容默认对下一轮 LLM 不可见，导致模型上一轮其实已经在 reasoning 中给出答案，但后续收尾仍可能输出空答案或错误答案。
2. 有些轨迹出现 step id 跳跃，主要来自 harness 内部 tool call、收尾重试或并发写入机制。
3. 工具调用容易出现搜索漂移：
   - 重复搜索相似关键词。
   - 被低质量 snippet 或 SEO 污染源带偏。
   - browser 403/500 后继续撞同一来源。
4. 最终答案容易发生槽位错误：
   - 问年份却输出解释。
   - 问公司或人物却输出泛称。
   - 问关系只输出“共”等不完整词。

这些问题推动了后续自进化模块设计。

为了避免影响原始 `track_d` 和用户另一个 `track_d_evo` 分支，新建了独立分支：

`track_d_slot_evo`

## 7. 反思模块设计

反思模块不是简单重跑，而是在失败后做结构化归因：

- `no_answer`：没有最终答案，后续要求尽早收尾并强制短答案。
- `tool_call_leak`：工具调用文本泄漏到答案，后续禁止伪工具调用。
- `uncertain_answer`：答案不确定，后续增加实体加槽位的 targeted query。
- `image_evidence_missing`：图像证据不足，后续使用图像线索加文本检索。
- `temporal_drift`：时间约束丢失，后续 query 必须带 first/before/after/as-of/year。
- `entity_drift`：实体漂移，后续先锁定图中实体再查属性。

PPT 可表述为：反思模块将失败轨迹转化为“失败类型、风险说明、修正策略”，用于后续任务的 prompt 注入和记忆更新。

## 8. 记忆模块设计

记忆以 JSONL 形式按 run 存储：

`runs/.../track_d_slot_evo/memory/evo_memory.jsonl`

每条记忆包含：

- 样本 index
- 任务类型
- 答案槽位
- 时间约束
- 失败类型
- slot failure
- 工具调用模式
- 重复 query 统计
- policy state
- 可复用 strategy
- pred preview
- token、turn、tool_calls、latency

检索逻辑按以下维度打分：

- task type 是否相同。
- answer slot 是否相同。
- 时间约束是否相交。
- 是否存在实体漂移、时间漂移、图像证据缺失等相似失败。
- 是否有工具调用模式和策略可复用。

重要约束：记忆不保存 gold 答案，只保存策略和失败经验，避免泄漏验证集答案。

## 9. 策略层与 Skills

当前 skills 包括：

| Skill | 作用 |
| --- | --- |
| `entity_lock` | 图像题先锁定图中实体，不轻易切换到搜索热门实体 |
| `temporal_lock` | 保留 first/before/after/as-of/season 等时间约束 |
| `slot_answer` | 最终答案只填目标槽位 |
| `visual_direct_answer` | 空间、类别、关系等视觉直答题优先看图，不被网页结果带偏 |
| `visual_fact_anchor` | 图像事实题先提取可见锚点，再查目标属性 |
| `cross_check_candidate` | 多候选时用判别 query 交叉验证 |
| `tool_failure_recovery` | browser 失败后换源或用 snippet，不反复撞失败页面 |

策略层会检测：

- 低可信源：PDF 聚合、答案站、SEO 污染、标题语义不匹配。
- 重复 query：相同或近似搜索意图。
- 时间约束遗漏：问题有 before/after/as-of，但 query 没带。
- 工具失败 fallback：浏览器错误后是否切换来源。

## 10. 关键实验结果

### 10.1 2wiki 全量结果

| 设置 | count | strict acc | LLM acc | avg tokens | avg turns | avg tools | avg latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| no-evo baseline | 100 | 0.42 | 0.74 | 29788.03 | 5.11 | 4.43 | 37.93 |
| slot evo v6 | 100 | 0.53 | 0.79 | 23821.22 | 4.60 | 3.79 | 34.96 |

2wiki 结论：

- LLM acc 提升：0.74 到 0.79，提升 5 个百分点。
- strict acc 提升：0.42 到 0.53，提升 11 个百分点。
- token、推理轮数、工具调用数、耗时均下降。
- 这是当前最适合 PPT 展示的正向自进化结果。

### 10.2 SimpleVQA 全量结果

| 设置 | count | strict acc | LLM acc | avg tokens | avg turns | avg tools | avg latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| no-evo baseline | 99 | 0.2828 | 0.6566 | 31374.32 | 5.70 | 4.45 | 67.23 |
| slot evo v6 | 99 | 0.3131 | 0.6364 | 26043.36 | 4.90 | 3.38 | 49.13 |
| slot evo v7 full fast | 99 | 0.3232 | 0.6566 | 33033.85 | 5.17 | 3.90 | 63.95 |

SimpleVQA 结论：

- strict acc 有提升：baseline 0.2828 到 v7 0.3232。
- LLM acc 与 baseline 持平：0.6566。
- v6 在效率上更好，但 LLM acc 低于 baseline。
- v7 小样本 30 条曾达到 LLM acc 0.7667，但全量后没有保持，说明 SimpleVQA 上存在过拟合或样本波动。

### 10.3 SimpleVQA v7 小样本

| 设置 | count | strict acc | LLM acc | avg tokens | avg turns | avg tools | avg latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| slot evo v7 l30 | 30 | 0.3667 | 0.7667 | 26625.40 | 5.13 | 3.67 | 46.53 |

该结果说明视觉实体锁定、视觉直答、答案槽位约束在局部样本有效，但全量泛化仍需要继续优化。

## 11. 典型失败分析

对 SimpleVQA 和 Track-D 轨迹观察到的失败模式：

1. 图像实体识别错误：
   - 例如 fortress 题，模型可能答 `Inkerman`，但正确应是具体堡垒名。
2. 图搜候选漂移：
   - search image 返回相似但错误实体，后续文本搜索围绕错误实体放大。
3. 答案槽位不干净：
   - 年份题输出完整解释。
   - 地点题输出国家或网页来源。
   - 类别题输出过长说明。
4. 视觉关系题过度搜索：
   - 本来可以看图回答，却被 web 搜索结果带偏。
5. 工具层不稳定：
   - `uguu` 图片上传偶发 502。
   - Jina 或 browser 偶发 timeout/503。
6. 并发和有状态进化冲突：
   - 开启 evolution updates 时，框架会把 stateful track 并发降为 1。
   - 若要快速全量验证固定策略，应使用 `--disable-evolution-updates`。

## 12. Benchmark 与验证命令要点

快速验证固定策略时使用：

```bash
python -u -m shared_sii_adapter.run_dataset \
  --input /inspire/qb-ilm2/project/26summer-camp-01/26210300/datasets/simpleVQA/SimpleVQA.jsonl \
  --track track_d_slot_evo \
  --output-dir /inspire/qb-ilm2/project/26summer-camp-01/26210300/runs/slot_evo_simplevqa_v7_full_fast \
  --group-id 8 \
  --llm-base-url "<Qwen3.5-9B服务URL>" \
  --model Qwen3.5-9B \
  --judge-base-url "<Qwen3-32B judge服务URL>" \
  --judge-model /inspire/qb-ilm2/project/26summer-camp-01/26210300/Qwen3-32B \
  --run-mode eval \
  --disable-evolution-updates \
  --concurrency 20 \
  --max-steps 20 \
  --max-tokens 16000 \
  --score-mode llm \
  --judge-concurrency 16
```

完整逐题进化时不要关闭 memory/evolution updates，但会自动串行运行，耗时更长。

## 13. 对评分项的对应关系

### 反思模块评分

已有实现点：

- 失败后触发 heuristic judge。
- 输出 failure type 和 reflection hint。
- 根据 trajectory 分析失败原因。
- 自动生成 repair strategy。
- 写入 debug 和 memory，供后续任务使用。

### 记忆模块评分

已有实现点：

- JSONL 长期记忆。
- 结构化字段包括任务类型、答案槽位、失败类型、工具模式、重复 query、策略。
- 后续任务按 task type、slot、constraint 相似度检索记忆。
- 注入 prompt 时明确只复用策略，不复制旧答案。

### 进化效率评分

已有可展示结果：

- 2wiki 上 evo 比 baseline 准确率更高。
- 2wiki 上 token、轮数、工具调用和耗时均下降。
- SimpleVQA 上 strict acc 有提升，但 LLM acc 未提升，需要说明多模态图像实体识别仍是主要瓶颈。

## 14. PPT 可用结论

1. 当前系统已经从“LLM 加工具加 prompt”扩展为轻量 agentic policy layer。
2. 自进化不是简单重跑，而是把失败轨迹沉淀为可复用策略。
3. 在 2wiki 文本多跳任务上，自进化显著提高准确率并降低工具成本。
4. 在 SimpleVQA 多模态任务上，视觉实体识别仍是主要瓶颈，策略对局部样本有效，但全量泛化有限。
5. 后续优化重点应放在：
   - 更强的图像实体识别。
   - 图搜候选交叉验证。
   - 视觉题 direct answer 与 web search 的动态选择。
   - 更严格的最终答案槽位清洗。
