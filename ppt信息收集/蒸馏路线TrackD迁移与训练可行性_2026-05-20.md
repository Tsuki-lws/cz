# 蒸馏路线 Track-D 迁移与训练可行性信息收集

更新时间：2026-05-20

## 1. 本轮探索背景

当前目标是把原本用于 benchmark 的 `track_d_ttl` 工具调用 Agent，迁移到蒸馏路线中，形成一套可以稳定产出教师轨迹、过滤轨迹、格式化 SFT/DPO 数据、再训练 Qwen3.5-9B 学生模型的工程链路。

核心设想：

- 教师模型：
  - 文本教师：Qwen3-32B
  - 视觉教师：Qwen2.5-VL-32B-Instruct
- 学生模型：
  - Qwen3.5-9B
- 目标：
  - 教师轨迹必须来自和 benchmark 一致的 Track-D harness。
  - 蒸馏后的学生模型再用同一套 Track-D harness 跑 benchmark。
  - 避免“训练轨迹格式”和“推理时 harness”不一致。

## 2. 为什么要复制 Track-D 到蒸馏路径

原 `track_d_ttl` 是 benchmark 路线的 Track-D 实现，里面包含：

- Track-D system prompt
- 工具调用约束
- 结构化 final answer 收尾
- 简单 heuristic judge
- reflection retry
- online memory
- 真实 `search_text` / `search_image` / browser 工具调用轨迹

直接修改原 `track_d_ttl` 风险较高：

- 会影响当前 benchmark 和历史可复现实验。
- prompt 或工具约束一旦改坏，会影响正式提交。
- 蒸馏需要额外字段、数据导出、teacher/student 标记，不应该污染原 track。

因此更合理做法是：

```text
原始 benchmark 分支：
track_d_ttl/

蒸馏专用分支：
distill/track_d_distill/
```

`distill/track_d_distill/` 可以从 `track_d_ttl/` 复制而来，再做蒸馏专用改造。

## 3. 蒸馏版 Track-D 应该改什么

蒸馏版不应该改核心答题能力，而是增加数据生产能力。

建议保留：

- 原 Track-D prompt 主体
- 每一步只能二选一：真实工具调用或最终答案
- 禁止伪工具调用
- 禁止空输出
- 收尾 JSON 格式：`{"final_answer":"..."}`
- `search_text` / `search_image` / browser 工具
- reflection retry 逻辑

建议新增：

- `teacher_name`
- `teacher_type`: text / vision
- `student_model`
- `seed_dataset`
- `source_split`
- `has_image`
- `trajectory_quality`
- `tool_error_count`
- `search_error_count`
- `browser_error_count`
- `final_answer_format_ok`
- `distill_keep`: 是否保留为训练样本
- `distill_reason`: 保留或剔除原因

蒸馏输出建议分三层：

```text
episodes.jsonl        # 每道题一条，包含最终答案、metrics、完整 records
trajectories.jsonl    # 展开后的逐 step 轨迹
sft_messages.jsonl    # 训练用 ShareGPT / messages 格式
```

## 4. 当前已经验证过的事情

### 4.1 Track-D harness 可以作为蒸馏轨迹来源

之前已经把 `distill/data_collection/teacher_collect.py` 改成调用 Track-D 路线，而不是旧的 `distill.harness.ReActAgent`。

当前教师收集逻辑：

```text
seed -> seed_to_task -> track_d_ttl.agent.run_one -> build_episode -> episodes.jsonl / trajectories.jsonl
```

这说明教师轨迹和 benchmark 使用的 Track-D 轨迹格式基本一致。

### 4.2 蒸馏旧 prompt 已经对齐 Track-D 输出风格

旧的 `distill/harness/prompts.py` 原先强制输出：

```text
<answer>...</answer>
```

这和 Track-D benchmark 路线不一致。已调整为：

- 最终答案直接输出短答案。
- 如果 harness 明确要求 JSON 收尾，才输出：

```json
{"final_answer":"你的答案"}
```

并且字段内要求：

- 只写最短答案
- 不解释
- 不写 Markdown
- 不写引用
- 不写 `<answer>` 标签
- 不写工具调用文本

## 5. benchmark 异常低分的定位

曾经用 student Qwen3.5-9B 跑过 `benchmark.csv`，和 `group_8_reference_answers.csv` 对比，结果很低：

```text
exact:      6 / 100
normalized: 6 / 100
loose:     10 / 100
llmjudge:  12 / 100
```

一开始看起来像是 prompt 或学生模型能力问题，但后续对比轨迹发现主要原因是工具服务异常。

### 5.1 坏结果的工具状态

student 那次运行中：

```text
search_text 错误率: 692 / 711 = 97.3%
总 tool 错误率:     1185 / 1475 = 80.3%
```

典型错误：

```text
ConnectionError: HTTPConnectionPool(host='127.0.0.1', port=1227)
Failed to establish a new connection: [Errno 111] Connection refused
```

说明 `search-proxy` 没有在 `127.0.0.1:1227` 正常运行。

### 5.2 历史较好结果的工具状态

历史较好的 Track-D 运行中：

```text
search_text 错误率: 7 / 1336 = 0.5%
总 tool 错误率:     99 / 1631 = 6.1%
```

这说明高分和低分之间的最大差异，不是 prompt，而是工具可用性。

### 5.3 关键结论

对工具调用 Agent 来说，benchmark 分数高度依赖：

- 搜索代理是否在线
- 浏览器服务是否在线
- 搜索结果是否稳定
- 并发是否压垮 search/browser 服务
- 模型是否能正确利用工具返回的证据

因此跑蒸馏或 benchmark 前，必须先做工具健康检查。

## 6. 工具健康检查建议

运行 Track-D 前必须检查：

```bash
ss -ltnp | grep -E ':1227|:8080'
```

期望：

```text
1227 search-proxy 监听中
8080 browser sandbox 监听中
```

再做一次真实搜索：

```bash
python - <<'PY'
from shared_sii_adapter.tools import search_text
print(search_text("Marine Products Corporation stock buyback shares remaining", top_k=1, fetch=False))
PY
```

如果这里报 `Connection refused`，不能开始跑 benchmark，也不能开始采集教师轨迹。

## 7. 训练速度判断

用户问题：已有数据集的情况下，蒸馏训练会不会快？

结论：如果数据集已经准备好，训练本身会比教师轨迹采集快很多；真正慢的是“教师生成轨迹”和“工具调用”。

### 7.1 最慢阶段

最慢的是教师轨迹收集：

```text
seed -> teacher model -> 多轮工具调用 -> 搜索 / 浏览器 -> reflection -> 保存 trajectory
```

原因：

- 每个样本可能 5-20 轮交互。
- 每轮可能调用搜索或浏览器。
- 搜索代理和浏览器服务有并发瓶颈。
- 32B / VL 教师模型生成也慢。

如果并发太高，例如 100，容易出现：

- search-proxy 压力过大
- browser 500
- 模型端点超时
- 部分样本中途失败

更稳的并发建议：

```text
teacher collection: 4-16
benchmark eval:     3-20
纯训练:             取决于 GPU 和 LoRA/全参设置
```

### 7.2 训练阶段相对更快

如果已经有干净的 SFT 数据，例如几百到几千条 messages：

- LoRA / QLoRA 会比较快。
- 9B 学生模型比 32B 教师生成轨迹快得多。
- 训练速度主要取决于：
  - GPU 数量
  - batch size
  - max sequence length
  - 是否训练多模态
  - 是否全参训练

粗略判断：

```text
几百条 SFT 样本：通常几十分钟到数小时
几千条 SFT 样本：通常数小时
上万条长轨迹样本：可能需要更久，且显存和序列长度会成为主要瓶颈
```

### 7.3 当前更推荐的路线

不要一开始就做很大的训练。

推荐顺序：

1. 先用 50-100 条轨迹验证格式。
2. 跑 `format_sft`，确认训练数据能被 LLaMA-Factory 读取。
3. 做一次小 LoRA smoke training。
4. 用 SimpleVQA 或小 benchmark 验证模型能加载。
5. 再扩大到 500-2000 条。
6. 最后再考虑 DPO / on-policy。

## 8. PPT 可以呈现的核心故事线

可以把本轮探索包装成：

```text
从 Agent Demo 到可蒸馏 Agent System
```

主要论点：

1. 原始 Track-D harness 已经具备工具调用、reflection、memory 和结构化答案收尾能力。
2. 如果直接用旧蒸馏 harness，会出现训练格式和 benchmark 推理格式不一致。
3. 因此需要复制 Track-D 到蒸馏路径，形成独立蒸馏 harness。
4. 低分实验定位出关键工程问题：工具服务不可用会让 Agent 从“检索增强”退化成“模型硬猜”。
5. 蒸馏数据质量不仅取决于 teacher 模型，还取决于工具链稳定性。
6. 后续训练应该先小规模闭环，再扩大数据。

## 9. 可放 PPT 的对比表

### 9.1 工具可用性对分数影响

| 运行版本 | search_text 错误率 | 总工具错误率 | loose | llmjudge | 结论 |
| --- | ---: | ---: | ---: | ---: | --- |
| student 异常运行 | 97.3% | 80.3% | 10% | 12% | 搜索代理挂了，模型基本在硬猜 |
| 历史较好 Track-D | 0.5% | 6.1% | 33%-37% | 约 40%+ | 工具可用时明显提升 |

### 9.2 蒸馏流水线阶段

| 阶段 | 输入 | 输出 | 主要风险 |
| --- | --- | --- | --- |
| seed loader | HotpotQA / TriviaQA / A-OKVQA 等 | seed_pool.jsonl | 数据集下载、字段不统一 |
| teacher collect | seed_pool + teacher endpoints | episodes / trajectories | 工具服务不稳定、teacher 慢 |
| filter | teacher episodes | filtered.jsonl | 错误答案混入训练集 |
| format SFT | filtered trajectories | ShareGPT / messages | 格式和 harness 不一致 |
| SFT train | SFT 数据 + Qwen3.5-9B | student checkpoint | 显存、seq_len、过拟合 |
| eval | student checkpoint + Track-D harness | benchmark / validation score | 工具状态和采样波动 |

## 10. 下一步建议

### 10.1 工程改造

新建：

```text
distill/track_d_distill/
```

从：

```text
track_d_ttl/
```

复制：

```text
agent.py
judge.py
refine.py
memory_online.py
```

然后把蒸馏收集脚本改为：

```python
from distill.track_d_distill.agent import run_one
```

这样可以避免修改原始 Track-D。

### 10.2 数据生产

先跑小规模：

```text
HotpotQA: 50
TriviaQA: 50
A-OKVQA: 50
```

然后检查：

- 轨迹是否有真实工具调用
- search error 是否过高
- final answer 是否干净
- teacher prediction 是否明显错误

### 10.3 训练

先做：

```text
SFT smoke: 50-100 条
正式 SFT v1: 500-2000 条
DPO / on-policy: 放到后面
```

## 11. 当前状态判断

当前蒸馏路线的关键不是“立刻训练”，而是先保证：

```text
Track-D 蒸馏 harness 独立化
teacher trajectory 稳定生成
工具服务健康检查
训练数据格式和 benchmark harness 一致
```

有数据集以后，训练本身不会是最慢的环节。真正需要重点控制的是教师轨迹质量和工具链稳定性。
