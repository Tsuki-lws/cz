# 关键实验记录归档

更新时间：2026-05-20

本文归档适合进入 Git 的轻量实验记录。完整 `runs/` 目录包含大量轨迹、日志、提交包和中间产物，体积较大且已被 `.gitignore` 排除；本文件只保存关键配置、结果数字、路径和结论，便于后续 PPT、README 和答辩复盘。

## 1. 归档原则

- 只提交轻量 Markdown 摘要，不提交完整 `runs/`。
- 保留关键实验路径，方便本地环境中追溯原始 summary 和轨迹。
- 对正式 benchmark、验证集、自进化消融和 oracle 诊断分开记录。
- 对不适合正向宣传的负结果也保留，便于说明探索过程和后续优化方向。

## 7. 可用于 PPT 的最终取舍

建议展示：

- Track-D benchmark 历史最佳正常版：30% strict / 44% LLM judge。
- 2wiki Slot Evo：42%/74% -> 53%/79%，同时降低 token 和工具调用。
- lightweight dedupe evolution：作为效率优化消融，展示 token 和 latency 降低。
- simpleVQA：展示 strict 小幅提升，同时说明视觉实体识别仍是瓶颈。
- oracle merge：只作为潜力诊断，不作为真实策略。

不建议夸大：

- 不说所有 evolution 都稳定提升准确率。
- 不把去重早停版说成准确率提升。
- 不把 oracle merge 当作真实 benchmark 分数。
- 不说 Qwen3-32B 直接参与作答。

## 8. 后续优化方向

1. 强化 final answer slot 校验，避免输出 `Wikipedia`、`I now have...`、`答案`、`无` 等非答案。
2. 对 2wiki 放宽必要证据深度，避免早停损伤多跳推理。
3. 对 simpleVQA 增强视觉实体锁定，减少图搜候选漂移。
4. 将长期经验离线总结为更稳定的 policy，而不是每条样本都写入泛化性弱的 lesson。
5. 从正确轨迹构造 SFT 数据，从多 run 正误轨迹构造 DPO pair。
6. 引入搜索和浏览缓存，降低复跑波动。
