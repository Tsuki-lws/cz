## 5. 多模态数据的重要发现

`benchmark.csv` 的情况已经确认：

1. 总计 100 条数据。
2. 其中 50 条是纯文本题。
3. 另外 50 条是多模态题。
4. 多模态题的 `image` 字段不是图片 URL，而是 **裸 base64 字符串**。
5. base64 图片内容中同时存在：
   - PNG
   - JPEG
   - WebP

这意味着：

1. 不能假设图片字段是 URL。
2. 不能直接把 `image` 当作 `search_image` 的输入 URL。
3. 需要优先让模型直接看图，再决定是否额外调用工具。

## 6. 当前验证结论

### 已验证可用

1. 浏览器服务可访问。
2. 搜索代理可访问。
3. 模型服务可访问。
4. 文本 smoke test 可在 `pegp` 环境中跑通。
5. 多模态图片已经能够传入模型，不再被静默忽略。

### 当前仍需优化的点

1. 多模态题虽然能把图片传给模型，但现有 prompt / tool 策略还不够稳。
2. 模型有时会因为“没有在线图片 URL”而绕去做文本搜索。
3. 原始 harness 偶尔会自造不存在的工具名，需要收紧 prompt 或 tool 约束。
4. 目前还没有完成正式的 benchmark 批量运行脚本。

## 12. 验证集当前完整结果

完整结果目录：

- `harness-sii/validation_full`

完整压缩包：

- `validation_full_results.tar.gz`：完整 validation_full、脚本等，约 76M。
- `validation_submit_bundle.tar.gz`：精简提交包，包含四套结果 jsonl、四套轨迹 jsonl、记忆文件、脚本和说明，约 26M。

四套结果：

- `2wiki_baseline`：100 条，57 条非空预测；本地 `answer` 字段为空，无法计算准确率。
- `2wiki_evolved`：100 条，45 条非空预测；100 条都有 32B Judge，45 条 Judge pass；本地 `answer` 字段为空，无法计算准确率。
- `simplevqa_baseline`：99 条，63 条非空预测；别名归一后 `31/99 = 31.31%`。
- `simplevqa_evolved`：99 条，51 条非空预测；99 条都有 32B Judge，69 条 Judge pass；别名归一后 `31/99 = 31.31%`。

结论：

- 当前 evolved 已合规接入 Judge/Memory，但没有超过 baseline。
- 主要瓶颈是 9B 的多模态识别和空输出。
- 工具模式在小样本上有一定收益，但速度更慢，后续建议分段继续试。

## 13. 后续推荐方向

1. 继续基于验证集优化，不要用打榜数据做自进化。
2. 优先优化 SimpleVQA，因为它有 gold 可本地评分。
3. 尝试 SimpleVQA 工具模式：允许 image_url 图搜，建议分段跑 `--start N --limit M`。
4. 对空输出样本做单样本内 Judge 触发重试，但不能把 gold 传给 Judge。
5. 优化 Memory：让 32B 归纳“题型 -> 识别步骤 -> 何时图搜”的结构化规则，不要让 memory 包含测试集 gold answer。
6. 并发建议先用 `--workers 2 --batch-size 2`；过高并发可能让 9B 服务空输出或排队变慢。
7. 所有完整验证集重跑都加 `--resume`，防止重复已完成样本。
